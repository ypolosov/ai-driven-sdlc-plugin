# ADR-013: Категории инструментов как агностический интерфейс

## Контекст

Целевой проект использует разные конкретные инструменты SDLC.
Плагин не должен фиксировать имена продуктов в каркасе по принципу 3.
Hooks и агентам нужен стабильный интерфейс к любому инструменту категории.
Существует прецедент `state_artifact.kind: file|dir|mcp` (ADR-004).
Принцип 13 уже привёл MCP как авторитативный backend для альф (ADR-009).

## Решение

Введены 7 категорий инструментов в `catalogs/tool-categories.md`.
Каждая категория — id, purpose, capabilities, binds_to_alphas.
Конкретные MCP-серверы регистрируются в `<target>/.mcp.json`.
Связь «категория → MCP» живёт в `<target>/.claude/sdlc/tool-bindings.md`.
Hooks и агенты ссылаются только на id категории.

## Последствия

- Плагин не зависит от конкретных продуктов целевого.
- Расширение через `tool-categories-extensions.md` без правки каркаса.
- Унифицированный контракт capabilities для всех инструментов категории.
- Скрипт `check-tool-binding.sh` отказывает при категории вне каталога.
- Сложность: контракт capabilities должен покрываться MCP-сервером целевого.

## Альтернативы

1. Жёсткая привязка к конкретному MCP — отвергнуто, нарушает принцип 3.
2. Универсальный «MCP-агент без типизации» — отвергнуто, теряется проверяемость.
3. Каталог продуктов в плагине — отвергнуто, лавинный рост поверхности.
