# Конституция плагина `ai-driven-sdlc`

## Назначение

Плагин — методологический каркас для разработки любой IT-системы.
Плагин не навязывает технологии, фреймворки и инструменты.
Плагин помогает выбрать уровень сложности и инструменты на каждой фазе SDLC.
Принятые решения фиксируются как артефакты Claude Code в целевом проекте.
Плагин применим к любой целевой системе, в том числе к самому себе.

## Ключевые определения (по Левенчуку)

### Альфа (alpha)
Объект отслеживания состояния в методе. Определения в `catalogs/alphas.md`.

### Дисциплина (discipline)
Знаниевый корпус, поддерживающий исполнение метода. Определения в `catalogs/disciplines.md`.

### Метод (method)
Схема работы: дисциплина + роль + рабочий продукт + инструмент.

### Роль (role)
Функциональная позиция актора в проекте. Определения в `catalogs/roles.md`.

### Рабочий продукт
Материальное свидетельство продвижения альфы.

### Целевая система
Система, на которую сейчас направлено внимание пользователя.

### Граф создания
Отношения между целевой системой и системой-создателем (см. Том 2 гл. 8).

## Принципы Волны 1

### 1. Интерактивность и фиксация альтернатив

Плагин всегда порождает 2–3 альтернативы и фиксирует выбор.
В HITL показывает пользователю и ждёт выбора.
В HOTL показывает и ожидает подтверждения.
В HOOTL пишет альтернативы в `decisions.md` без предъявления.

### 2. Ситуативная инженерия методов (SME)

Уровень сложности — pet / mid / enterprise — выбирается по каждой фазе независимо.
Выбор фиксируется в `target/.claude/sdlc/profile.md`.
Уровень может меняться по мере развития проекта.

### 3. Абстрактное в плагине, конкретное в целевом

Плагин несёт только каркас: альфы, дисциплины, методы, роли, мета-шаблоны, матрицу.
Конкретные шаблоны инструментов инстанцируются в целевом проекте.
Запрещены шаблоны типа Lean Canvas, arc42, ADR, User Story, TODO.md внутри плагина.
Обоснование: Том 1 гл. 4 «Воплощение и описание»; Том 1 гл. 5; Том 2 гл. 10.

### 4. Русский язык и ≤15 слов на утверждение

Все артефакты пишутся на русском языке.
Каждое утверждение содержит не более 15 слов.
Структура артефакта соответствует мета-шаблону строго.
Исключение 1: блоки цитат в fenced-кавычках не подпадают под правила.
Исключение 2: отчёт `audit.md` имеет собственный мета-шаблон без ≤15 слов.

### 4a. Код без комментариев

В сгенерированном коде не пишутся комментарии.
Код — единственный источник истины о своём поведении.
Документация живёт в markdown-артефактах, не в коде.
Исключения: shebang, license header, pragma, type directive.

### 5. TDD по умолчанию с мягкой блокировкой

PreToolUse hook на пути исходников требует подтверждения при отсутствии парного теста.
Scope и пары source↔test конфигурируются в `plugin-config.md`.
Второй слой — LLM-аудитор проверяет семантику теста относительно diff.
Третий слой — coverage-gate как fitness-функция фазы testing.

### 6. Детерминированное приоритетнее недетерминированного

Любая задача, решаемая скриптом или хуком, не делегируется LLM.
Форматеры и линтеры обязательны для фазы development.
Hook-диспетчер не знает языков; команды читаются из `plugin-config.md`.
После прохождения development без форматера hook блокирует запись кода.

### 7. Граница целевой системы определяется вниманием

Пользователь переносит внимание на любой уровень системной иерархии.
Плагин помогает определить относительно выбранной системы: надсистему, подсистемы, окружение, систему создания.
Результат фиксируется в `target/.claude/sdlc/system-context.md`.
Обоснование: Том 2 гл. 7 «Системные уровни»; Том 2 гл. 9.

### 8. Выбор автономности на задачу

Три уровня автономности: HITL, HOTL, HOOTL.
Дефолт устанавливается на уровне фазы в `profile.md`.
Per-task override задаётся командой `/sdlc-autonomy --task <level>`.
Per-task override ephemeral и не сохраняется в файлах.

### 9. Состояние проекта в артефактах целевого

Плагин не держит внутреннего состояния о целевом проекте.
Состояние читается из state-артефакта, указанного пользователем.
Имя и тип state-артефакта фиксируются в `plugin-config.md`.
Плагин не навязывает формат: плоский файл, каталог, GitHub Issues через MCP.

### 10. Credentials в `.env` целевого проекта

Секреты хранятся в `.env` в корне целевого проекта.
`.env` обязательно добавлен в `.gitignore`.
Плагин поставляет `.env.example` и правило `.gitignore`.

### 11. Ролевая навигация

Плагин спрашивает, какую роль пользователь играет сейчас.
Плагин предлагает фазы и задачи по роли и состоянию альф.
Каталог ролей — `catalogs/roles.md`, с полем `interests` (Том 1 гл. 6).

### 12. Применим к любому IT-продукту, включая себя

Плагин может быть применён к любому IT-продукту.
В том числе к самому плагину как целевой системе (dogfooding).
Обоснование: Том 2 гл. 8 «Графы создания».

### 13. Критический аудит и консистентность

Отдельная роль `sdlc-consistency-auditor` обеспечивает сквозную согласованность артефактов.
Аудит запускается PostToolUse (async) и вручную через `/sdlc-audit`.
Единственный источник истины о состоянии альф — `sdlc-alpha-tracker`.
Другие агенты обращаются к трекеру, не читают `alphas.md` напрямую.

### 14. Конвенция пути артефактов SDLC

Артефакты SDLC живут в `<target>/.claude/sdlc/`.
Hooks и агенты опираются только на этот префикс.
Имя state-артефакта может быть вне `.claude/`; фиксируется в `plugin-config.md`.
`/sdlc-init` поддерживает режимы: `--fail-if-exists` (по умолчанию), `--merge`, `--force`.

### 15. Эволюция принципов через memom

`CLAUDE.md` плагина — источник истины о текущих принципах.
`memom.md` плагина — журнал изменений; не источник правды.
Формат записи: дата, формулировка до/после, мотив, последствия.
Изменение принципа в `CLAUDE.md` без записи в `memom.md` блокируется pre-commit hook.

### 16. README плагина как живой артефакт

`ai-driven-sdlc-plugin/README.md` актуализируется по мере развития.
Содержит инвентарь (имена и счётчики) skills, commands, agents, scripts, meta-templates.
При изменении публичной поверхности — README обновляется в том же коммите.
`check-readme-inventory.sh` сверяет имена в README со структурой; переименование без обновления README блокируется.

### 17. README per система внимания в целевом

Каждая система внимания (`/sdlc-focus`) получает описание по `system-readme.meta.md`.
Две категории:
- `materialized` → `<system-root>/README.sdlc.md` (sidecar, не трогает существующий README).
- `logical` → `<target>/.claude/sdlc/external-systems/<slug>.md`.
Флаги `/sdlc-focus`: `--transient` (без описания), `--retire` (архивирование в `retired-systems/`).
TTL из `plugin-config.md`: после N дней без фокуса — приоритет расхождений понижается.
Политика для существующего корневого `README.md` целевого — **refuse**; плагин не трогает его.

## Правила работы с артефактами

### Где живут артефакты
Все артефакты SDLC создаются в целевом проекте в `<target>/.claude/sdlc/`.
Внутри плагина артефакты SDLC никогда не создаются.
Исключение: сам плагин в dogfooding-режиме становится целевым.

### Обязательные файлы в целевом
- `<target>/.claude/CLAUDE.md` — конституция целевого проекта.
- `<target>/.claude/sdlc/profile.md` — SME-выбор по фазам.
- `<target>/.claude/sdlc/plugin-config.md` — конфиг hooks.
- `<target>/.claude/sdlc/alphas.md` — состояние альф.
- `<target>/.claude/sdlc/system-context.md` — фокус внимания.
- `<target>/.claude/sdlc/roles.md` — играемые роли.
- `<target>/.claude/sdlc/decisions.md` — журнал альтернатив.
- `<target>/.env.example` и `<target>/.gitignore` (с записью `.env`).

### Что плагин не создаёт внутри себя
Плагин не хранит конкретных шаблонов: Lean Canvas, arc42, ADR, User Story, TODO.md.
Они инстанцируются в целевом по мета-шаблонам + `context7`.

## Внешние зависимости

### MCP-серверы
- `context7` — подтягивание референсной структуры выбранных пользователем инструментов.
- `github` — опционально, для state-артефакта через GitHub Issues на mid-уровне.

### Инструменты-спутники
- `plugin-dev` — валидация структуры плагина.

## Работа над самим плагином

### GitHub Flow
Разработка плагина идёт по GitHub Flow: feature-ветка → PR → review → merge в main.
Каждая публичная поверхность (skill, command, agent, script, принцип) — одна ветка.

### Правила коммитов
Коммиты на русском языке, conventional commits допустим.
В коммитах нет комментариев к коду; диффы самодокументируемы.
