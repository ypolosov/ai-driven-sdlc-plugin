# Skill: установка уровня автономности

## Назначение фазы

Задаёт режим взаимодействия с пользователем: HITL / HOTL / HOOTL.
Фазовый дефолт записывается в `profile.md`.
Per-task override живёт только в текущей сессии.

## Альфы, которые продвигает

- **Way of Working** — уточнение режима взаимодействия с командой.

## Дисциплины

- Situational Method Engineering.

## Уровни автономности

- **HITL (Human in the loop)** — подтверждение на каждый шаг.
- **HOTL (Human on the loop)** — наблюдение, остановка по запросу.
- **HOOTL (Human out of the loop)** — полная автономия, альтернативы логируются.

## Мета-вопросы

1. Какую фазу настраиваем (или «все фазы сразу»)?
2. Какой уровень автономности выставить по умолчанию?
3. Устанавливать per-task override для текущей задачи?
4. Если HOOTL: подтверждено ли, что критические действия согласованы заранее?

Конкретные инструменты автономности не выбираются.
Принцип 1 применяется всегда: альтернативы порождаются и фиксируются в `decisions.md`.

## Матрица SME

Не применяется напрямую; выбор уровня — методологическое решение.
Ориентир: pet и mid чаще используют HITL/HOTL, enterprise с автотестами — HOOTL на детерминированных участках.

## Протокол инстанцирования

1. Прочитать текущий `profile.md`.
2. Для указанной фазы обновить `default_autonomy` (или для всех фаз).
3. При per-task флаге `--task` — выставить ephemeral в текущей сессии, не писать в файл.
4. Зафиксировать изменение в `decisions.md` (принцип 1).
5. Hooks в следующий раз считают новое значение автоматически.

## Критерии готовности

- `profile.md` содержит валидное значение автономности для затронутых фаз.
- `decisions.md` содержит запись с мотивом смены.
- При HOOTL на опасных фазах (deployment, operations) — явное подтверждение в decisions.

## Следующие шаги по ролям

- method-engineer → проверить соответствие автономности и выбранных инструментов.
- любая другая роль → продолжить работу: плагин автоматически применит новый режим.
